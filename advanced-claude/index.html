Part 1: The Power User Mindset

Claude Code is an operating system for AI-assisted development. Most people use 10% of it.

Right now, you open Claude Code, type what you want, and Claude builds it. That works. But you're leaving massive power on the table.

Think of it this way: you've been driving a sports car in first gear. This guide teaches you to use all the gears. Here's what most people never discover:

  • 60+ slash commands that control everything from models to themes to session management
  • Plan mode where Claude researches and plans without changing anything, then you approve
  • Subagents — parallel workers that do research, run tests, and explore code simultaneously
  • Custom agents you design yourself with specific roles and permissions
  • Hooks that automatically run shell commands every time Claude edits a file, makes a commit, or takes any action
  • MCP servers that connect Claude to GitHub, Slack, databases, APIs, and any external service
  • CLAUDE.md files that permanently teach Claude your project's rules, patterns, and preferences
  • Memory files where Claude records what it learns across sessions so it never forgets
Your Setup Already Uses Some of This
Your CLAUDE.md already has rules for book projects, gig outreach, and app conventions. Your memory files track service worker versions, localStorage keys, and app architectures. But there's SO much more you can do with these systems.

Part 2: All 60+ Slash Commands

Type / in the prompt to see a filterable list. Every command explained below — what it does, when you'd reach for it, and real examples.

Session & Context Management

These are the commands you'll use most. They control your conversation's memory, let you jump between projects, and keep Claude from forgetting what you're doing.

/clear

Wipes the entire conversation and starts fresh. Claude forgets everything — what files it read, what changes it made, what you were building. The session itself stays (you can scroll up), but Claude's working memory is gone. Also available as /reset or /new.

When to use it: You've finished one task and want to start something completely unrelated. You were fixing a Zercher bug, now you want to work on the book shelf. There's no reason for Claude to remember the workout code. 

/clear

When NOT to use it: If you might come back to what you were doing. Use /rename first, then /clear. Or better — just open a new terminal tab.

/compact

Compresses the conversation to free up context space. Claude summarizes everything that's happened so far into a condensed form, then continues with room to work. You can add focus instructions in quotes to tell Claude what to prioritize keeping.

When to use it: Long sessions where Claude starts slowing down, repeating itself, or forgetting earlier decisions. Building a 1,500-line book HTML is the classic case — you're on chapter 15 and Claude forgot the CSS structure from chapter 1. 

/compact "keep the HTML structure,
  chapter list, and which chapters are done"

The focus instruction matters. Without it, Claude compresses everything equally and might drop the details you care about. With it, Claude knows to preserve your HTML progress and drop the earlier back-and-forth about cover images.

Pro tip: You can compact preemptively. Don't wait until Claude is confused — compact after every major milestone (every 3-4 chapters of a book build, after finishing a big feature).

/resume

Picks up a previous session right where you left off. Claude reloads the full conversation — every message you sent, every file it read, every change it made, every decision that was discussed. It's like you never left. Also available as /continue.

The full lifecycle: Say you're working on the Advanced Claude guide right now. It's getting late and you want to pick this up tomorrow. Here's exactly what you do: 

/rename advanced-claude-guide

That names this session. Now close the terminal, shut the laptop, go to bed. Tomorrow, open your terminal and type: 

claude --resume advanced-claude-guide

Claude loads up with the full conversation intact. It remembers you were editing Part 2, it knows what files were changed, it picks up right where you left off. You can just say "let's keep going with the next section" and it knows what you mean.

If you don't remember the exact name, or you didn't name it, just type /resume by itself. Claude shows a scrollable list of all your recent sessions — named ones show their name, unnamed ones show an auto-generated summary. Pick the one you want.

When to use it: Any time you're leaving a session you might want to come back to. Working on a book HTML and it's bedtime. Fixing a Bloom bug but need to switch to Zercher first. You can have multiple named sessions saved and jump between them like browser tabs. 

/resume bloom-splitter-fix
/resume book-road-to-serfdom
/resume zercher-plate-calc

What gets restored: The full conversation history, your working directory, and Claude's understanding of the project. What does not get restored: background processes that were running, or the exact state of your terminal output.

Pro tip: Always /rename before leaving a session. Without a name, you'll be scrolling through sessions labeled "2026-03-19 session" trying to remember which one had your Bloom work. With a name, you just type /resume bloom-splitter and you're back instantly.

/branch

Forks the conversation at this exact point. Both branches keep the full history up to now, but diverge from here. The original session continues untouched. Also available as /fork.

When to use it: You want to try two different approaches and see which works better. Maybe you're debating whether to refactor the Bloom splitter as a separate component or keep it inline. Branch, try approach A. If it doesn't work, go back to the original and try approach B. 

/branch try-component-approach

Key difference from /clear: /clear erases everything. /branch preserves everything and gives you a safe copy to experiment in. It's like git branching but for your conversation.

/rewind

Undoes Claude's work back to a previous point in the conversation. Shows you a list of checkpoints and lets you pick which one to revert to. This undoes both the conversation AND the file changes Claude made after that point. Also available as /checkpoint.

When to use it: Claude went down a wrong path. It refactored your service worker and broke cloud sync, or it changed 5 files when it should've changed 1. Instead of manually undoing everything, rewind to before it started. 

/rewind

Important: This actually reverts file changes on disk, not just the conversation. It's like a targeted git checkout for everything Claude touched since that checkpoint.

/rename

Gives the current session a human-readable name so you can find it later with /resume. Without a name, sessions get auto-generated IDs that are impossible to remember.

When to use it: Any time you're doing meaningful work you might want to come back to. Name it right away so you don't forget. 

/rename zercher-plate-calc-fix
/rename book-road-to-serfdom
/rename bloom-stock-picks-feature

Pro tip: Use descriptive names with the app name first. When you /resume later and see the picker list, "bloom-stock-picks-feature" tells you exactly what that session was. "fix-bug" tells you nothing.

/context

Shows a visual grid of how much context (memory) Claude is using. Each colored block represents a chunk of the conversation. When the grid is mostly full, Claude is running out of room and will start forgetting things.

When to use it: Claude starts acting confused, repeating itself, or forgetting instructions you gave earlier. Check /context to see if it's a memory problem. If the grid is 80%+ full, time to /compact. 

/context

/btw

Asks a quick side question without adding it to the conversation history. Claude answers your question, but the exchange doesn't take up context space or distract from the main task.

When to use it: You're deep into building a crypto backtest and need to quickly check what toFixed(2) does, or how Array.reduce() works. You don't want Claude to lose track of the backtest just because you asked a JavaScript question. 

/btw what does toFixed(2) do?
/btw how do I center a div vertically?
/btw what's the git command to see
  the last 5 commits?

/exit

Closes Claude Code. Also available as /quit. Same as pressing Ctrl+C twice or Ctrl+D. Your session is auto-saved — you can /resume it later.

Model & Performance

These commands let you switch between AI models and tune how hard Claude thinks. Different tasks need different levels of brainpower.

/model

Switches which AI model Claude uses. The three main options: Opus (smartest, slowest, most expensive), Sonnet (balanced), Haiku (fastest, cheapest, less capable). Changes take effect immediately for the next message.

When to use it: You're doing something complex (planning an architecture, writing nuanced book HTML) and want Opus. Or you're doing something simple (renaming variables, formatting) and Haiku is plenty. 

/model opus
/model sonnet
/model haiku

Pro tip: Start complex tasks on Opus for the planning phase, then switch to Sonnet for the repetitive implementation. You get the best thinking upfront without burning through your quota on mechanical work.

/effort

Controls how deeply Claude thinks before responding. Low gives quick, shallow answers. High makes Claude reason more carefully. Max is the deepest thinking mode (Opus only, session-only — doesn't persist). Auto resets to default.

When to use it: You're debugging a tricky cloud sync race condition — set effort to high so Claude really thinks through the timing. Or you're just asking Claude to add a CSS class — low effort is fine. 

/effort high
/effort low
/effort max

/fast

Toggles fast mode on or off. When on, Claude uses cheaper, quicker processing. Responses come faster but may be less thorough.

When to use it: You're doing a bunch of small, simple tasks in a row — renaming files, adding console.logs, quick lookups. Fast mode saves time and cost. 

/fast on
/fast off

Code & Git

/diff

Opens an interactive diff viewer showing every uncommitted change in the project. You can see exactly what Claude modified, line by line, file by file. Use arrow keys to navigate between files. Left/right switches between the full git diff and per-turn diffs (what changed in each Claude response).

When to use it: Before committing. Claude just made a bunch of changes to Roman Chariots' receipt scanner and you want to review every line before pushing. Or Claude changed something unexpected and you want to see what. 

/diff

Pro tip: Get in the habit of running /diff before every commit. It catches things Claude might have changed that you didn't ask for — extra imports, reformatted code, removed comments.

/plan

Puts Claude into plan mode. In this mode, Claude can read files, search code, and explore the project, but it cannot make any changes. It can only propose changes for you to approve. Think of it as read-only reconnaissance.

When to use it: Before starting a big change. You want Claude to understand how the Bloom growth chart works before modifying it. Or you want a plan for adding a new feature to the gig outreach app before Claude starts writing code. 

/plan

Plan mode is covered in depth in Part 3.

/pr-comments

Fetches comments from a GitHub pull request and displays them in your session. Auto-detects the PR for your current branch, or you can pass a PR number or URL. Requires the gh CLI to be installed.

When to use it: Someone left review comments on your PR and you want Claude to see them and help address the feedback. 

/pr-comments
/pr-comments 42

/security-review

Analyzes all pending changes on your current branch for security vulnerabilities. Looks for injection risks, auth issues, exposed credentials, data leaks, and other OWASP top-10 problems.

When to use it: Before merging a PR that touches user input, API endpoints, or authentication. Good habit for any change that handles data from outside your app. 

/security-review

Configuration

Settings, appearance, and permissions. Most of these you'll set once and forget.

/config

Opens a settings interface where you can change your model, theme, output style, and other preferences. A one-stop shop for all settings. Also available as /settings.

When to use it: When you want to change multiple settings at once, or you're not sure which specific command to use. Instead of remembering /theme vs /model vs /effort, just open /config and browse. 

/config

/theme

Changes the color theme of Claude Code. Options include light, dark, and several colorblind-accessible variants (daltonized). Also ANSI terminal palettes if your terminal has custom colors. 

/theme

/color

Sets the color of your prompt bar for the current session. Purely cosmetic but useful if you run multiple Claude sessions in different terminal tabs — color-code them so you know which is which at a glance. 

/color red
/color purple
/color default

Available colors: red, blue, green, yellow, purple, orange, pink, cyan.

/permissions

Shows what Claude is currently allowed to do and lets you update it. Covers file editing, bash commands, network access, and other tool permissions.

When to use it: Claude keeps asking for permission and you want to pre-approve certain actions. Or you want to lock things down for a sensitive task — restrict Claude to read-only while you explore. 

/permissions

/vim

Toggles Vim editing mode for the input prompt. When on, you can use Vim keybindings (hjkl, i for insert, Esc for normal mode, etc.) while typing your messages to Claude.

When to use it: If you're a Vim user and your muscle memory keeps hitting Esc and j/k when writing prompts. 

/vim

/terminal-setup

Configures terminal keybindings for Shift+Enter (multi-line input) and other shortcuts. Only appears in terminals that need it (VS Code integrated terminal, Alacritty, Warp). If your terminal already handles these keys natively, you won't see this command. 

/terminal-setup

/keybindings

Opens or creates the keybindings config file at ~/.claude/keybindings.json. Lets you rebind keyboard shortcuts, add chord bindings, or change the submit key. 

/keybindings

/statusline

Configures the status line that appears at the bottom of Claude Code. You can describe what you want to see (context usage, model name, token count) or run it without arguments to auto-configure from your shell prompt. 

/statusline

Project & Memory

These commands manage how Claude remembers things across sessions — your project's CLAUDE.md, auto-memory, and working directories.

/init

Creates a CLAUDE.md file in your project root. This file gives Claude persistent instructions — project structure, coding conventions, build commands, things to always remember. Claude reads it at the start of every session.

When to use it: Starting a brand new project. You want Claude to always know that "this is a single-file HTML PWA that uses fireGet() for writes" or "always bump the service worker version when pushing." 

/init

/memory

Opens your memory files for editing. These are the CLAUDE.md files (project-level and user-level) plus auto-memory entries that Claude creates as it learns about your codebase. You can enable/disable auto-memory, edit entries, or delete things Claude remembered incorrectly.

When to use it: Claude keeps forgetting something important (like always stripping commas from price strings) — add it to memory so it sticks. Or Claude learned something wrong and you need to correct it. 

/memory

/add-dir

Adds another directory to the current session's scope. By default, Claude can only see files in the directory where you launched it. This command expands that.

When to use it: You started Claude in the Bloom directory but now need to reference the SPX dashboard's Apps Script code too. 

/add-dir ../SPX
/add-dir ~/Documents/Code/Claude/Books

Real Scenarios: Putting It Together

Scenario: Building a Book HTML (Lessons of History)

You're 2 hours into building a book's index.html. It's going great but you're at 3,000 lines and Claude is getting slow. Here's the power user move: 

/rename book-lessons-history
/compact "keep the HTML structure,
  chapter list, and where we left off"

Now you've named the session (so you can resume it tomorrow if needed) and compressed the context so Claude has room to keep going. Without /compact, the session would've hit the context wall and lost track of what chapters were done.

Scenario: Switching Between Apps

You're working on Bloom's DCA splitter and get a bug report about Zercher's plate calculator. Instead of losing your Bloom context: 

/rename bloom-splitter-fix

Then start a new session for Zercher. When you're done with the Zercher fix: 

claude --resume bloom-splitter-fix

You're right back where you left off with Bloom, full context intact.

Scenario: Trying Two Approaches

You're adding a recommendation engine to the gig outreach app. You could score venues by distance, or by similarity to past gigs. You want to try both: 

/branch distance-approach

Try the distance-based scoring. If it feels wrong, go back to the original session and try similarity-based instead. Both branches are saved — you can compare them later.

Scenario: "Wait, Undo That"

You asked Claude to refactor the cloud sync in Roman Chariots, but it changed the JSONP fetch pattern that was working fine. Files are already modified on disk. 

/rewind

Pick the checkpoint from before the refactor. Claude reverts all the file changes back to that point. No need to manually undo anything.

Scenario: "Did I Break Something?"

Claude just made changes across 4 files in the Sora podcast app and you're not sure what all changed before committing: 

/diff

Opens an interactive viewer showing every file changed, every line added and removed. You review it all before doing git commit.

Scenario: Quick Question Mid-Task

You're deep into building the crypto backtest and need to know what the toFixed() function does, but you don't want to distract Claude from the main task: 

/btw what does toFixed(2) do
  in JavaScript?

Gets an instant answer without polluting the main conversation about the backtest.

Quick Reference
Lost? Type / and start typing to filter. You don't need to memorize all of these. The five you'll use daily are /compact, /rename, /resume, /diff, and /clear. Everything else is there when you need it.

Part 3: Plan Mode Deep Dive

Look before you leap. Plan mode lets Claude research and design before touching code.

Plan mode is read-only. Claude can explore your codebase, search files, read code, and think — but it cannot edit, write, or run commands. This is perfect when you want Claude to understand something before changing it.

How to Enter Plan Mode

  • During a session: Press Shift+Tab to cycle through permission modes until you see "plan mode on"
  • Via command: Type /plan
  • At startup: claude --permission-mode plan

When to Use Plan Mode

  • New features: "How would you add dark mode to Cadence?" — let Claude explore the codebase and propose an approach before writing code
  • Refactoring: "How should I restructure the cloud sync?" — Claude maps all the pieces first
  • Bug investigation: "Why does the splitter sometimes show $0?" — Claude traces the logic without accidentally breaking things
  • Architecture decisions: "Should I use a service worker or just cache headers?" — Claude researches trade-offs

The Plan Mode Workflow

Real Example From Your Work
When building a new book HTML, Claude enters "plan mode" mentally — reading notes.md, checking the template, understanding the chapter structure — before writing a single line of HTML. You could formalize this by starting in plan mode, reviewing the proposed structure, then switching to execution.

Real Scenarios: Plan Mode in Action

Scenario: Redesigning the Bloom Splitter

The weekly DCA splitter needs a new feature: redirect money from portfolios that are at their hard cap to ones that aren't. This touches the splitter logic, the DCA algorithm, the UI, and the cloud sync.

Without plan mode (what you did before): "Hey Claude, add cap redistribution to the splitter." Claude dives in and starts editing. Maybe it misses that the growth chart also depends on splitter data. You find the bug later during testing.

With plan mode: 

/plan
"I want to add cap redistribution to the
Bloom splitter. Before you change anything,
explore how the splitter works, what depends
on it, and propose a plan."

Claude reads the splitter function, traces what calls it, finds the growth chart dependency, and presents a plan: "Here are the 4 functions I'd change, here's the order, and here's the risk with the growth chart." You review, tweak the approach, then Shift+Tab to execute.

Scenario: Adding Cloud Sync to a New App

You just built a new app and need to add the full cloud sync pattern (auto-push, auto-pull, fireGet, empty data guard). Plan mode first: 

/plan
"Look at how Cadence does cloud sync
and plan how to add the same pattern
to this app. Show me every function
I'll need."

Claude reads Cadence's sync code, understands the pattern, and writes you a plan with all 5 required pieces. You approve, it builds — no missed pieces, no data loss bugs.

Scenario: Investigating a Bug Without Breaking Things

The Crypto dashboard's 2-day timer resets randomly. You don't want Claude to start "fixing" things before you understand the root cause: 

/plan
"The 2-day timer in the crypto app
is resetting randomly. Investigate
the timer logic and tell me what
could cause premature resets."

Claude traces the timer code, reads the localStorage keys, checks the visibilitychange handler, and identifies that the cloud sync pull is overwriting the timer data. Now you know the real cause before touching anything.

Part 4: Subagents & Delegation

Spawn parallel workers that research, explore, and execute tasks independently.

Subagents are separate Claude instances that run inside your session. They have their own context window, their own tools, and they report back when done. Think of them as interns you can send on errands.

Built-in Subagent Types

  • Explore (uses Haiku — fast and cheap) — searches files, reads code, maps the codebase. Perfect for "find where X is defined" or "what files handle authentication?"
  • Plan (inherits your model) — researches during plan mode, gathers context before presenting options
  • General-purpose (inherits your model, all tools) — complex multi-step tasks. Can read, write, search, and run commands
  • Bash — runs terminal commands in isolation

When Claude Uses Subagents

Claude automatically delegates when tasks benefit from parallelism or isolation:

  • Research in parallel: "Read the Cadence sync code and the Bloom sync code and compare them" — two Explore agents run simultaneously
  • Keep main context clean: verbose outputs (test results, large file reads) go to a subagent so your main conversation stays focused
  • Compacting notes: for book projects, two agents compact the first and second half of notes in parallel

Creating Custom Subagents

This is where it gets powerful. You can create your own specialized agents with specific roles, tools, and permissions. Create a markdown file in .claude/agents/: 

# .claude/agents/code-reviewer.md
---
name: code-reviewer
description: Reviews code for quality
tools: Read, Glob, Grep
model: sonnet
maxTurns: 10
---

You are a senior code reviewer.
Check for:
- Security vulnerabilities
- Performance issues
- Missing error handling
- Inconsistent patterns

Now you can say "use the code-reviewer to check the Cadence sync function" and Claude spawns your custom agent.

Ideas for Custom Agents

  • test-runner: runs tests and reports only failures
  • docs-writer: generates documentation from code
  • css-auditor: checks for unused styles and inconsistencies
  • accessibility-checker: scans HTML for a11y issues
  • perf-analyzer: profiles JavaScript for bottlenecks

Background Tasks

Subagents can run in the background while you keep working. Press Ctrl+B to send a running task to the background. You'll be notified when it finishes.

Important Limitation
From your CLAUDE.md: NEVER use agents to write large HTML files. Agents consistently produce ~400 lines when 1000+ are needed. Agents are great for research, compacting notes, and running commands — but write final output directly.

Real Scenarios: Subagents at Work

Scenario: Book Project Note Compaction (What We Already Do)

For "The General Theory" (485 pages), bookai produced ~5,000 lines of raw notes. Too much for the HTML build. Here's how we use subagents: 

You: "Compact notes.md. Use two parallel
agents — first half and second half.
Target 1,500 lines total."

Two subagents run simultaneously. Agent 1 compacts chapters 1-12 (2,500 lines down to 750). Agent 2 compacts chapters 13-24 (2,500 lines down to 750). Both finish at the same time. Claude merges the results. Total time: ~2 minutes instead of ~4 minutes if done sequentially.

Scenario: Comparing Sync Patterns Across All Your Apps

Part 5: CLAUDE.md & Memory

The Instruction Hierarchy

Writing Effective CLAUDE.md Rules

Your current CLAUDE.md is already great. Here are principles that make it work:

  • Be specific: "ALWAYS bump service worker version in service-worker.js when pushing" is way better than "Don't forget about caching"
  • Use action words: "NEVER use agents to write HTML" is clear. "Agents aren't great at HTML" is vague.
  • Include the WHY: "Use fireGet() for writes (POST silently fails due to CORS redirects)" — Claude understands the reason and can apply it to new situations
  • Keep it under 200 lines: Longer CLAUDE.md = less adherence. Move detailed stuff to separate files

Auto Memory

Memory is where Claude stores what it learns across sessions. When Claude hits a bug and figures out the fix, it records that insight so it doesn't repeat the mistake.

Your memory file already tracks things like:

  • Service worker versions for each app
  • localStorage keys and their purposes
  • App architectures and sync patterns
  • Past bugs and their solutions
  • Your preferences and workflow patterns

Memory lives at ~/.claude/projects/<project>/memory/MEMORY.md. Claude can create topic-specific files like debugging.md or patterns.md and link to them from the main MEMORY.md.

Part 6: Hooks — Automate Everything

Hooks are shell commands that fire automatically when Claude takes actions. They're your safety net and automation layer.

Imagine if every time Claude edited a file, it automatically ran a code formatter. Or every time it tried to touch a protected file, it got blocked. Or every time it finished a task, it sent you a desktop notification. That's what hooks do.

Hook Events

Hooks fire at specific moments in Claude's workflow:

  • PreToolUse — before Claude uses a tool (edit, write, bash). You can block it.
  • PostToolUse — after a tool runs. Great for formatting, linting, testing.
  • SessionStart — when a session begins or resumes after compaction
  • Notification — when Claude needs your attention (permissions, idle)
  • Stop — when Claude finishes responding

Practical Hook Examples

1. Auto-Format After Every Edit

// In .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "prettier --write \"$FILE\""
        }]
      }
    ]
  }
}

2. Desktop Notification When Claude Needs You

{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [{
          "type": "command",
          "command": "osascript -e 'display notification \"Claude needs attention\" with title \"Claude Code\"'"
        }]
      }
    ]
  }
}

3. Block Edits to Protected Files

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "if echo \"$FILE\" | grep -q '.env'; then exit 2; fi"
        }]
      }
    ]
  }
}

Exit code 2 blocks the action. Exit code 0 allows it. The error message from stderr gets sent back to Claude as feedback.

Hook Types
Command — runs a shell command (most common).
Prompt — asks an AI to decide (yes/no with reasoning).
Agent — spawns a subagent to inspect and decide.
HTTP — sends event data to a web endpoint (for logging/audit).

Part 8: Permissions & Security

Control exactly what Claude can and cannot do.

Permission Modes

  • Default: asks before running commands or editing files
  • Accept Edits: auto-approves file changes, still asks for bash commands
  • Plan: read-only, no changes allowed
  • Don't Ask: auto-denies unless pre-approved in settings
  • Bypass: (--dangerously-skip-permissions) skip all prompts. Only for trusted automation.

Cycle through modes with Shift+Tab during a session.

Permission Rules

You can pre-approve or block specific actions in your settings: 

// In ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Bash(git push)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Read(./.env)",
      "Edit(./.env)"
    ]
  }
}

Deny always wins. If both allow and deny match, the action is blocked.

Claude runs everything without asking. Use this only when you trust the workflow (like your outreach scripts that you've already tested). Never use it on a new codebase you don't understand.

You're tired of approving git push every single time. Add it to your allowed list:

Part 9: Context Window Mastery

The context window is your session's memory. Managing it well = longer, better sessions.

Claude has a 200,000-token context window. That sounds huge, but it fills up faster than you think — every file Claude reads, every command output, every message back and forth consumes tokens.

How to Monitor Context

When context gets full, Claude auto-compacts — it summarizes the conversation so far to free up space. You can trigger this manually anytime:

The optional instruction tells Claude what to prioritize in the summary. Everything else gets compressed more aggressively.

  • Use subagents for verbose output: "Use a subagent to run all the tests and just tell me what failed" — the full test output stays in the subagent's context, not yours
  • Compact before big tasks: If you've been chatting and are about to start a big build, /compact first to free up space
  • Name your sessions: /rename book-serfdom so you can resume exactly where you left off instead of re-establishing context
  • Keep CLAUDE.md lean: under 200 lines. Move details to linked files that load on demand
The Output Token Limit
Remember that error you saw? API Error: exceeded the 32000 output token maximum. You can increase it:

Think of custom commands as saved prompts. Every time you find yourself typing the same complex instruction, turn it into a command. Over time, you build a library of one-word shortcuts for your entire workflow. /push-all, /book-start, /sync-check — each replaces 5+ lines of instructions.
You find yourself typing "commit, bump service worker, push, cachebust" after every change. What's the power user move?
A custom command turns a repetitive multi-step workflow into a single /push-all. CLAUDE.md is for rules (things Claude should always do). Custom commands are for workflows (things you want to trigger on demand). Both are useful, but commands are the right tool here.

Terminal Power Moves

These aren't Claude Code features — they're terminal tricks that make your Claude sessions smoother. The kind of things power users just do automatically.

caffeinate: Keep Your Mac Awake

When Claude is in the middle of a big task — building a 1,300-line book HTML, running a batch of note compactions, processing a PDF — the last thing you want is your Mac going to sleep and killing the session. 

# Wrap Claude in caffeinate
caffeinate claude

# Or if Claude is already running,
# caffeinate in another terminal tab:
caffeinate -d -i -s

caffeinate is a built-in macOS command that prevents sleep. When you run caffeinate claude, your Mac stays awake for the entire Claude session and goes back to normal sleep behavior when you exit. The flags:

  • -d — prevent display sleep (screen stays on)
  • -i — prevent idle sleep (system stays awake)
  • -s — prevent system sleep (even on battery)
You're Already Doing This
Look at your terminal tab right now — it says "caffeinate" in the title. That's this command running, keeping your Mac awake while we build this guide.

tmux / screen: Persistent Sessions

What if your terminal closes accidentally? tmux keeps your session alive even if you close the terminal window: 

# Start a persistent session
tmux new -s claude-work

# Run Claude inside tmux
caffeinate claude

# If your terminal closes, reattach:
tmux attach -t claude-work
# Claude is still running!

This is like giving your Claude session an insurance policy. Terminal crash? SSH disconnect? Just reattach and everything is still there.

Multiple Terminal Tabs

Use separate terminal tabs for different concerns:

  • Tab 1: caffeinate claude (your main session)
  • Tab 2: cd Books && python3 -m http.server (preview your book HTML locally)
  • Tab 3: git commands, file browsing, ADB cachebust commands

Troubleshooting Cheat Sheet

  • CLAUDE.md not being followed? Run /memory to verify it loaded. Check for conflicting rules. Make instructions more specific.
  • Context running out? /compact "focus on X". Move verbose work to subagents. Keep CLAUDE.md under 200 lines.
  • MCP server not connecting? /mcp to check status. Verify environment variables. Try claude mcp list.
  • Hook not firing? /hooks to verify config. Check matcher is exact (case-sensitive). Verify correct event name.
  • Permissions too strict? /permissions to view rules. Deny rules override allow rules.
