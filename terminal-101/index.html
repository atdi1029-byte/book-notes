Part 1: What Is the Terminal?

It's just a text-based way to talk to your computer. That's it.

Every time you open Terminal on your Mac and type caffeinate claude, you're using the terminal. It looks intimidating — black screen, blinking cursor, no buttons — but it's doing the same things your Finder does. Just with text instead of clicking.

GUI vs Terminal

You already know two ways to interact with your computer:

  • GUI (Graphical User Interface) — clicking icons, dragging files, using menus. Finder, Chrome, VS Code.
  • Terminal (Command Line Interface) — typing text commands. Faster for many things, the only way to do others.

Same actions, different interface. Terminal is just faster for certain things.

The Prompt

When you open Terminal, you see something like this: 

alexbarnett@mac ~ %

That's the prompt. It's telling you:

  • alexbarnett — who you are (your username)
  • @mac — which computer
  • ~ — where you are (~ means your home folder, /Users/alexbarnett)
  • % — "ready for input" (zsh uses %, bash uses $)

After the prompt, you type a command and press Enter. The computer runs it and shows you the result. That's the entire interaction model.

Shells: bash vs zsh

The shell is the program that reads your commands and executes them. Your Mac uses zsh (Z Shell) by default. An older shell called bash is also available. They're 95% identical for everyday use. The main differences:

  • zsh — macOS default since 2019. Config file: ~/.zshrc. Prompt ends with %.
  • bash — the classic. Used in most scripts. Config file: ~/.bashrc. Prompt ends with $.
Why This Matters
When Claude tells you to "add this to your .zshrc," it's telling you to put a setting in your shell's configuration file. That file runs every time you open a new terminal tab, so anything in it is always available.
You see alexbarnett@mac ~/Documents % in your terminal. What does ~/Documents mean?
The ~ is shorthand for your home directory (/Users/alexbarnett ). So ~/Documents means you're currently in /Users/alexbarnett/Documents. The prompt always shows where you are.

Part 2: Navigating the File System

Moving around your files and folders from the command line.

Your Mac's file system is a tree. Everything starts at the root (/) and branches out into folders. Here's the part you use: 

/Users/alexbarnett/
  Documents/
    Code/
      Claude/
        Books/          <-- book notes project
        Bloom/          <-- DCA portfolios
        Cadence/        <-- task app
        Zercher/        <-- workout tracker
        SPX/            <-- stock dashboard
        Crypto/         <-- crypto dashboard
        Email/          <-- gig outreach
        Sora/           <-- podcast app
        RomanChariots/  <-- budgeting app

The Essential Navigation Commands

pwd: "Where Am I?"

$ pwd
/Users/alexbarnett/Documents/Code/Claude/Books

pwd stands for "print working directory." It tells you exactly where you are. Think of it as checking the map.

ls: "What's Here?"

$ ls
Bloom/  Books/  Cadence/  Crypto/  Email/
RomanChariots/  SPX/  Sora/  Zercher/

# Show more details
$ ls -la
drwxr-xr-x  12 alex  staff  384 Mar 19 10:30 Bloom/
-rw-r--r--   1 alex  staff  93K Mar 19 10:15 index.html
drwxr-xr-x   8 alex  staff  256 Mar 18 14:22 Zercher/

ls lists what's in the current folder. The -l flag shows details (permissions, size, date). The -a flag shows hidden files (names starting with a dot, like .git). We'll decode those permission letters in Part 6.

cd: "Go There"

# Go into a folder
$ cd Documents/Code/Claude/Bloom

# Go up one level
$ cd ..

# Go up two levels
$ cd ../..

# Go home (from anywhere)
$ cd ~

# Go to the previous directory
$ cd -

cd stands for "change directory." It's like double-clicking a folder in Finder, except you type the name instead of clicking.

Path Types: Absolute vs Relative

There are two ways to specify a location:

  • Absolute path — starts with /, the full address. Like a street address: /Users/alexbarnett/Documents/Code/Claude/Bloom
  • Relative path — starts from where you are now. Like saying "two blocks north": ../Bloom (go up one folder, then into Bloom)
# These do the same thing if you're
# in /Users/alexbarnett/Documents/Code/Claude:

# Absolute (full address)
$ cd /Users/alexbarnett/Documents/Code/Claude/Bloom

# Relative (from current location)
$ cd Bloom

Special Shortcuts

  • ~ — your home directory (/Users/alexbarnett)
  • . — the current directory (used in ./discover.sh to mean "run this file right here")
  • .. — one directory up (the parent folder)
  • - — the last directory you were in (toggle back)
Tab Completion
Start typing a file/folder name and press Tab. The terminal autocompletes it. If there are multiple matches, press Tab twice to see all options. This saves SO much typing and prevents typos. Use it constantly.
You're in ~/Documents/Code/Claude/ Bloom and want to get to Zercher. What's the shortest command?
cd ../Zercher goes up one folder (from Bloom to Claude/) then into Zercher. Relative paths are almost always shorter. The absolute path works too, but it's way more typing.

Part 3: Working with Files

Creating, copying, moving, and deleting files from the command line.

Creating Files and Folders

# Create a new empty file
$ touch notes.md

# Create a new folder
$ mkdir my-new-app

# Create nested folders in one shot
$ mkdir -p Books/new-book/images
# -p means "create parent folders too"

Copying and Moving

# Copy a file
$ cp index.html backup.html

# Copy a folder (need -r for "recursive")
$ cp -r Bloom/ Bloom-backup/

# Move (also used for renaming)
$ mv old-name.html new-name.html

# Move a file to another folder
$ mv notes.md ../Books/general-theory/

mv does double duty: it moves files AND renames them. If the destination is a new name in the same folder, it's a rename. If it's a different folder, it's a move.

Deleting (Be Careful!)

# Delete a file
$ rm notes.txt

# Delete a folder and everything inside
$ rm -r old-project/

# The nuclear option (DANGEROUS)
$ rm -rf folder/
# -r = recursive, -f = force (no prompts)
# This deletes EVERYTHING with no undo
No Trash Can
Unlike dragging to Trash in Finder, rm is permanent. There is no undo, no trash folder, no recovery. Files are just gone. That's why Claude's CLAUDE.md says "NEVER run rm -rf unless explicitly asked." This command has destroyed more data than any bug in history.

Reading Files

# See the whole file
$ cat index.html

# See just the first 20 lines
$ head -20 index.html

# See just the last 20 lines
$ tail -20 service-worker.js

# Count lines in a file
$ wc -l index.html
2782 index.html

You've seen Claude use wc -l all the time to check how many lines a file is. That's "word count" with the -l (lines) flag.

Finding Files

# Find files by name
$ find . -name "*.html"
./index.html
./plain.html
./dca.html

# Find files containing text
$ grep -r "fireGet" .
./Bloom/index.html:function fireGet(url) {
./Cadence/index.html:function fireGet(url) {
./Zercher/index.html:function fireGet(url) {

find searches by filename. grep searches by content. When Claude says "let me search the codebase," it's essentially running grep across your files.

What does cp -r Bloom/ Bloom-backup/ do?
cp -r copies recursively — the folder and everything inside it. The original stays untouched. -r is needed because cp alone only copies single files, not folders.

Part 4: Pipes & Redirection

The | and > symbols that make the terminal truly powerful.

This is where the terminal goes from "text-based Finder" to something genuinely more powerful than any GUI. Pipes let you chain commands together, passing the output of one into the input of the next.

The Pipe: |

The pipe (|) takes the output of one command and feeds it as input to the next. Think of it as an assembly line: 

# Without pipe (two steps):
$ ls -la > temp.txt
$ grep ".html" temp.txt

# With pipe (one step, no temp file):
$ ls -la | grep ".html"
-rw-r--r--  1 alex  staff  93K index.html
-rw-r--r--  1 alex  staff  12K plain.html

ls -la lists all files. The pipe sends that list to grep ".html" which filters it down to only lines containing ".html".

Real Pipes from Your Workflow

# Count how many HTML files you have
$ find . -name "*.html" | wc -l
47

# See the 5 biggest files
$ ls -lS | head -5

# Find all apps using fireGet
$ grep -rl "fireGet" . | grep index.html
./Bloom/index.html
./Cadence/index.html
./Zercher/index.html

# Check git log in a readable format
$ git log --oneline | head -10

Pipes chain commands: list files → filter to .html → count them → 47

Redirection: > and >>

Instead of showing output on screen, you can redirect it to a file: 

# Write output to a file (overwrites!)
$ echo "Hello" > message.txt

# Append to a file (adds to the end)
$ echo "More stuff" >> message.txt

# Save a command's output
$ git log --oneline > git-history.txt

# Redirect errors to a file
$ npm install 2> errors.txt

> overwrites the file. >> appends to it. This is important: if you use > on a file that exists, the old content is gone.

Command Chaining: && and ;

# Run second command ONLY if first succeeds
$ git add . && git commit -m "update"
# If git add fails, commit won't run

# Run second command regardless
$ cd Bloom ; ls
# Even if cd fails, ls runs anyway

# Chain multiple commands
$ git add . && git commit -m "fix" && git push
# All three must succeed in sequence

&& is "and then, if OK." ; is "and then, no matter what." You see Claude use && constantly when chaining git commands.

What does grep -r "toast" . | wc -l do?
grep -r "toast" . finds all lines containing "toast" in all files. The pipe sends those results to wc -l which counts the lines. So you get a number: how many times "toast" appears across your codebase.

Part 5: Environment Variables

Settings that control how your terminal and programs behave.

Environment variables are like global settings for your terminal. Programs read them to know how to behave. You've already used them without knowing it.

What Are They?

# See a variable's value
$ echo $HOME
/Users/alexbarnett

$ echo $USER
alexbarnett

$ echo $SHELL
/bin/zsh

# See ALL environment variables
$ env

The $ prefix means "the value of this variable." $HOME expands to /Users/alexbarnett. That's why ~ works as a shortcut — it's actually $HOME.

Setting Variables

# Set a variable for this session only
$ export MY_NAME="Alex"
$ echo $MY_NAME
Alex

# Set for just one command
$ CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000 claude

# Make it permanent (add to ~/.zshrc)
$ echo 'export CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000' \
  >> ~/.zshrc

PATH: The Most Important Variable

$PATH is the single most important environment variable. When you type a command like git, the shell doesn't magically know where the git program lives. It searches through a list of folders, and that list is $PATH. 

$ echo $PATH
/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin

That's a list of folders separated by colons. When you type git, the shell checks each folder in order:

  1. Is git in /opt/homebrew/bin/? Yes! Run it.
  2. If not, check /usr/local/bin/
  3. If not, check /usr/bin/
  4. If not found anywhere: "command not found"
"Command Not Found" Decoded
This error just means the program isn't in any of the folders listed in $PATH. Common fixes:
  • The program isn't installed (install it with brew install X or npm install -g X)
  • It's installed but not in PATH (add its folder to PATH in ~/.zshrc)
  • You're missing the ./ prefix for a local script (./discover.sh not discover.sh)

Variables You've Already Used

# Claude's output limit
export CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000

# Claude's effort level
export CLAUDE_CODE_EFFORT_LEVEL=high

# Which model to use
export ANTHROPIC_MODEL=claude-opus-4-6

# Auto-compaction threshold
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50

These all work the same way: you set a variable, and Claude Code reads it when it starts up. The program checks "does this variable exist?" and if so, uses its value to configure itself.

You type bookai and get "command not found." What's most likely the problem?
"Command not found" always means the shell can't find the program in any PATH folder. Either it's not installed, or you need to specify the full path to it (like ./bookai if it's in the current directory).

Part 6: Permissions & chmod

Why scripts need chmod +x and what those rwx letters mean.

Every file on your Mac has permissions that control who can do what with it. When you see -rwxr-xr-x in an ls -l listing, those letters are the permission code.

Reading Permission Codes

-rwxr-xr-x  1 alex  staff  discover.sh
drwxr-xr-x  8 alex  staff  Bloom/

That first column is 10 characters. Here's what each position means:

-rwxr-xr-x: file, owner can read/write/ execute, everyone else can read/execute

chmod: Changing Permissions

# Make a script executable
$ chmod +x discover.sh

# Now you can run it
$ ./discover.sh

chmod +x adds the execute permission. Without it, the file is just text. With it, the system treats it as a program that can run. This is the most common chmod you'll ever use.

When Claude tells you to run chmod +x script.sh, it's because you just created a new script file and it doesn't have execute permission yet. Without chmod +x, running ./script.sh gives you "Permission denied."

Why ./ for Scripts?

When you type discover.sh, the shell looks in $PATH folders for a program named "discover.sh." It won't find it because your current folder isn't in PATH.

./discover.sh says "run the file right here in this directory." The ./ is an explicit path, bypassing the PATH search entirely.

Common Pattern
New script workflow:
  1. Create the file: touch myscript.sh
  2. Write code in it
  3. Make it executable: chmod +x myscript.sh
  4. Run it: ./myscript.sh
Steps 3-4 are what you do with discover.sh and pipeline.sh.

Part 7: Shell Scripts Decoded

Reading the automation scripts Claude writes for you.

A shell script is just a text file full of terminal commands. Instead of typing them one by one, the script runs them all automatically. Your discover.sh and pipeline.sh are shell scripts.

Anatomy of a Script

#!/bin/bash                  # 1. Shebang
                             #    "Use bash"

MAX_GIGS=5                   # 2. Variables

echo "Starting discovery..." # 3. Commands

if [ -f "venues.json" ]; then  # 4. Conditions
  echo "Venues file exists"
else
  echo "No venues file!"
  exit 1                     # Stop with error
fi

for venue in "${venues[@]}"; do  # 5. Loops
  echo "Processing $venue"
  curl -s "$API_URL?q=$venue"
done

The Building Blocks

1. The Shebang: #!/bin/bash

The first line of every script tells the system which program should run it. #!/bin/bash means "use bash." #!/usr/bin/env python3 means "use Python." Without this line, the system doesn't know how to interpret the file.

2. Variables in Scripts

# Set a variable (no spaces around =!)
MAX_GIGS=5
VENUE_NAME="Bistrot Lepic"

# Use a variable
echo "Processing $MAX_GIGS gigs"
echo "Venue: ${VENUE_NAME}"

# Command-line arguments
# $1 = first argument, $2 = second, etc.
# ./discover.sh search "wineries VA"
# $1 = "search", $2 = "wineries VA"
Gotcha: No Spaces!
In bash, NAME="Alex" sets a variable. But NAME = "Alex" (with spaces) tries to run a command called "NAME" with arguments "=" and "Alex" and fails. No spaces around the equals sign!

3. Conditionals: if/then/fi

# Check if a file exists
if [ -f "notes.md" ]; then
  echo "Notes found"
fi

# Check if a variable is empty
if [ -z "$RESULT" ]; then
  echo "No results"
fi

# Check if a command succeeded
if git push; then
  echo "Push succeeded"
else
  echo "Push failed!"
fi

Notice the fi at the end — it's "if" backwards. That's how bash marks the end of an if block. (Yes, it's weird.)

4. Loops: for and while

# Loop through a list
for app in Bloom Cadence Zercher; do
  echo "Checking $app..."
  cd "$app" && git status && cd ..
done

# Loop through files
for file in *.html; do
  echo "$file has $(wc -l < "$file") lines"
done

# Loop while a condition is true
while [ $count -lt 10 ]; do
  echo "Attempt $count"
  count=$((count + 1))
done

5. Functions in Scripts

# Define a function
check_sw_version() {
  local file="$1/service-worker.js"
  if [ -f "$file" ]; then
    grep "CACHE" "$file" | head -1
  else
    echo "No service worker found"
  fi
}

# Call the function
check_sw_version "Bloom"
check_sw_version "Zercher"

Reading Your Discover Script

Now when you look at discover.sh, you can read it: 

#!/bin/bash
# ^ Use bash to run this

MAX_GIGS=${MAX_GIGS:-3}
# ^ If MAX_GIGS isn't set, default to 3

echo "Discovering venues from $MAX_GIGS gigs..."
# ^ Print a status message

for gig in "${past_gigs[@]}"; do
  # ^ Loop through each past gig
  osascript -e "tell application \"Chrome\"..."
  # ^ Use AppleScript to control Chrome
  sleep 5
  # ^ Wait 5 seconds (don't overload the API)
done

It's just a list of commands with some logic (loops, conditions) to control the flow. Nothing magical.

Part 8: Git Under the Hood

What branches, remotes, and commits actually are.

You use git every day through Claude. "Commit and push" is second nature. But understanding what git is actually doing makes you more confident and helps when things go wrong.

What Git Actually Is

Git is a time machine for your code. Every time you commit, git takes a snapshot of all your files. You can go back to any snapshot, compare snapshots, or create parallel timelines (branches).

The Three Stages

git add = "I want to save these changes." git commit = "Save them permanently." 

# See what's changed (working dir)
$ git status

# Stage specific files
$ git add index.html service-worker.js

# Commit (snapshot the staged changes)
$ git commit -m "Add buy alert feature"

# Push to GitHub (share the snapshot)
$ git push

What's a Commit?

A commit is a snapshot with metadata:

  • Hash — a unique ID like ae13b2c (the first 7 chars of a long hex string)
  • Author — who made it
  • Date — when
  • Message — why ("Add buy alert feature")
  • Parent — which commit came before it (forming a chain)
$ git log --oneline -5
ae13b2c Add Terminal Power Moves section
5b2b800 Add Headless Mode and Custom Commands
e6c5516 Add APIs, localStorage, Service Workers
4d0bbb3 Add Real Scenarios to all sections
586e279 Initial Advanced Claude guide

Remotes: Local vs GitHub

Your git repository exists in two places:

  • Local (on your Mac) — in the .git/ folder inside your project
  • Remote (on GitHub) — called origin
# See your remotes
$ git remote -v
origin  https://github.com/atdi1029-byte/
  book-notes.git (fetch)
origin  https://github.com/atdi1029-byte/
  book-notes.git (push)

# Push sends your local commits to GitHub
$ git push origin main

# Pull downloads GitHub's commits to local
$ git pull origin main

git push = upload. git pull = download. origin is just a nickname for the GitHub URL. main is the branch name.

Branches: Parallel Timelines

A branch is like a copy of your project that you can change independently. The main branch is called main (or sometimes master). 

# See what branch you're on
$ git branch
* main

# Create a new branch
$ git checkout -b experiment

# Switch between branches
$ git checkout main

# Merge a branch back into main
$ git checkout main
$ git merge experiment

This is what Claude's --worktree feature uses under the hood. Each worktree is a separate branch in its own directory.

The .git Folder

Every git repo has a hidden .git/ folder that contains ALL of git's data — every commit, every branch, every version of every file. 

$ ls .git/
HEAD        # what branch you're on
config      # remote URLs, settings
objects/    # all the file snapshots
refs/       # branch pointers

Never edit anything in .git/ manually. Git manages it. If you delete .git/, you lose all git history (but keep the current files).

What's the difference between git add and git commit?
git add moves changes to the staging area — "these are the ones I want to save." git commit takes everything in staging and creates a permanent snapshot. Neither one touches GitHub — that's git push.

Part 9: ADB Demystified

What those Android Debug Bridge commands actually do.

ADB (Android Debug Bridge) is how your Mac talks to your Android phone via USB. Claude uses it constantly to cachebust your apps after pushing changes. Let's decode what's actually happening.

The Basic Structure

adb -s R3CW506ZK7N shell "am start ..."
 |   |              |      |
 |   |              |      +-- Android command
 |   |              +-- "run on the phone"
 |   +-- YOUR phone's serial number
 +-- Android Debug Bridge

Every ADB command follows this pattern: adb (the program) + -s SERIAL (which phone) + shell (run on the phone) + the actual command to run.

Your Device Serial Number

R3CW506ZK7N is the unique ID of YOUR phone. If you ever get a new phone, this number changes and all the ADB commands in CLAUDE.md need updating. 

# See all connected devices
$ adb devices
List of devices attached
R3CW506ZK7N	device

If you see "no devices" or "unauthorized," the phone isn't connected properly or you need to approve USB debugging on the phone screen.

The Cachebust Command Decoded

adb -s R3CW506ZK7N shell \
  "am start \
   -a android.intent.action.VIEW \
   -d 'https://atdi1029-byte.github.io/\
bloom-portfolios/?cachebust=1710847200'"

Breaking this down:

  • am start — "Activity Manager, start something." It's like tapping an app icon.
  • -a android.intent.action.VIEW — "I want to VIEW something" (opens in Chrome)
  • -d 'URL' — the URL to open
  • ?cachebust=1710847200 — a random number (timestamp) appended to the URL so Chrome treats it as a brand new page and doesn't serve from cache

So the whole command means: "Hey phone, open Chrome and load this URL with a unique parameter so the service worker fetches fresh files instead of using the cache."

Other Useful ADB Commands

# Take a screenshot
$ adb -s R3CW506ZK7N exec-out \
  screencap -p > screenshot.png

# Install an APK
$ adb -s R3CW506ZK7N install myapp.apk

# See running apps
$ adb -s R3CW506ZK7N shell \
  pm list packages | grep chrome

# View device logs (debugging)
$ adb -s R3CW506ZK7N logcat | grep -i error
The Command You Must NEVER Run
adb shell pm clear com.android.chrome wipes ALL of Chrome's data — every cookie, every localStorage entry, every cached file. That means ALL your PWA apps (Cadence, Bloom, Zercher, Roman Chariots, Sora) lose their local data simultaneously. This happened on March 14 2026 and it's why CLAUDE.md has the rule: NEVER run pm clear. Always use cachebust instead.
You push an update to Bloom but your phone still shows the old version. What should you do?
Cachebust via ADB opens the URL with a fresh parameter, forcing Chrome to check for new files. NEVER clear Chrome data — that wipes localStorage for every app. Waiting might work eventually, but cachebusting is instant.

Part 10: Terminal Power Moves

The tricks that make everything faster.

caffeinate: Keep Your Mac Awake

When Claude is building a 1,300-line book HTML or processing a PDF, the last thing you want is your Mac going to sleep. caffeinate prevents that: 

# Keep awake while Claude runs
$ caffeinate claude

# Keep awake indefinitely (in another tab)
$ caffeinate -d -i -s
# -d = no display sleep
# -i = no idle sleep
# -s = no system sleep (even on battery)

# Keep awake for a specific time (2 hours)
$ caffeinate -t 7200

When you run caffeinate claude, it wraps Claude in a "stay awake" wrapper. When Claude exits, caffeinate stops and your Mac can sleep normally again.

Shell Aliases: Custom Shortcuts

An alias turns a long command into a short one. Add these to your ~/.zshrc: 

# Quick Claude
alias cc="caffeinate claude"
alias ccs="caffeinate claude \
  --dangerously-skip-permissions"
alias ccr="caffeinate claude --continue"

# Quick navigation
alias books="cd ~/Documents/Code/Claude/Books"
alias bloom="cd ~/Documents/Code/Claude/Bloom"
alias apps="cd ~/Documents/Code/Claude"

# Quick cachebust
bust() {
  adb -s R3CW506ZK7N shell \
    "am start -a android.intent.action.VIEW \
    -d '$1?cachebust=$(date +%s)'"
}

# Usage: bust "https://...github.io/bloom-portfolios/"

After saving ~/.zshrc, run source ~/.zshrc to reload it. Now cc starts Claude with caffeinate, books jumps to the Books folder, and bust URL cachebusts your phone.

History: What Did I Type Before?

# See command history
$ history

# Search history
$ history | grep "git push"
1234  git push origin main
1567  git push

# Re-run a previous command
$ !1234

# Re-run the LAST command
$ !!

# The up/down arrow keys scroll
# through recent commands

You can also press Ctrl+R and start typing to search through your command history interactively. This works both in the regular terminal AND inside Claude Code.

Multiple Tabs and tmux

Use multiple terminal tabs for different things:

  • Tab 1: caffeinate claude (main session)
  • Tab 2: local preview server or git commands
  • Tab 3: ADB commands and phone debugging

For even more power, tmux keeps your session alive even if the terminal window closes: 

# Start a named session
$ tmux new -s claude-work

# Detach (leave running in background)
# Press: Ctrl+B then D

# Reattach later
$ tmux attach -t claude-work
# Everything is still running!

Watching Files in Real-Time

# Watch a file's line count update live
$ watch -n 1 wc -l index.html
Every 1.0s: wc -l index.html
2782 index.html    # updates every second

# Get notified when a file changes
$ fswatch -o index.html | while read; do
  echo "Changed at $(date +%H:%M:%S)"
done

Quick Reference: Keyboard Shortcuts

  • Tab — autocomplete file/folder names
  • Ctrl+C — cancel the running command
  • Ctrl+L — clear the screen
