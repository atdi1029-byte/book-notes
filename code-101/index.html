Part 1: The Big Picture

Three languages work together to make every web page and app you use.

Every app Claude builds for you — Cadence, Bloom, Zercher, Roman Chariots, this guide you're reading right now — is made of exactly three things:

Every web app is made of these three pieces

Think of building a house. HTML is the framing — the walls, the rooms, where the doors go. CSS is the paint, wallpaper, and furniture — it makes everything look good. JavaScript is the electricity and plumbing — it makes things actually work when you flip a switch.

A Real Example: Your Cadence App

When you open Cadence on your phone, the browser reads one big file: index.html. Inside that file are all three languages stacked on top of each other: 

<!DOCTYPE html>        <!-- "This is a web page" -->
<html>
<head>
  <style>
    /* CSS goes here - colors, fonts, layout */
    --accent: #6ecfcf;     /* Cadence's teal */
    --orange: #e8944c;     /* The orange accent */
  </style>
</head>
<body>
  <!-- HTML goes here - buttons, text, cards -->
  <h1>Cadence</h1>
  <div class="task-card">Buy groceries</div>

  <script>
    // JavaScript goes here - logic, saving data
    function addTask(name) {
      data.tasks.push({ name: name });
      saveData();
    }
  </script>
</body>
</html>
Key Insight
All your apps are single-file apps. HTML, CSS, and JavaScript all live in one index.html file. That's why Claude can build entire apps in one shot — everything is in one place.

Why This Matters for Vibe Coding

You don't need to write code. But understanding what the three pieces do helps you:

  • Describe what you want better — "Change the CSS" vs "make the button bigger" both work, but knowing which is which helps you be precise
  • Understand what Claude changed — when Claude says "I updated the event handler," you'll know that's JavaScript, not styling
  • Debug faster — if something looks wrong, it's probably CSS. If something doesn't work, it's probably JavaScript. If something is missing, it's probably HTML.
Quick check: Your Zercher app's plate calculator shows the wrong number of plates. Which language is the bug probably in?
Correct! The plate calculator is logic — it calculates numbers. That's always JavaScript. CSS only affects how things look, and HTML only affects what's on the page.

Part 2: HTML — The Skeleton

Tags are containers. That's really the whole idea.

HTML is the simplest of the three languages. Everything in HTML is a tag — a container that wraps around content. Tags have an opening and closing: 

<h1>This is a heading</h1>
<p>This is a paragraph.</p>
<button>Click me</button>

That's it. Tags wrap content. The tag name tells the browser what kind of content it is. Here are the ones you'll see most often:

The Tags You'll See Everywhere

  • <div> — a generic box. Like a cardboard box — it can hold anything. Most of your apps are built from divs.
  • <p> — a paragraph of text
  • <h1> through <h4> — headings, from biggest to smallest
  • <button> — a clickable button
  • <input> — a text field the user can type in
  • <img> — an image
  • <a> — a link (the "a" stands for "anchor")
  • <span> — a small inline container, like wrapping a word in highlighter

Tags Inside Tags (Nesting)

Tags go inside other tags, like Russian nesting dolls. Here's a task card from Cadence: 

<div class="task-card">          <!-- outer box -->
  <div class="task-header">      <!-- inner box -->
    <span class="task-name">     <!-- text wrapper -->
      Buy groceries
    </span>
    <button class="delete-btn">  <!-- button -->
      X
    </button>
  </div>
</div>

The indentation shows you what's inside what. Each level of indent means "this is inside the tag above me." Claude always indents code this way — it's how you read the structure at a glance.

Attributes: Extra Info on Tags

Tags can have attributes — extra info inside the opening tag. The two most important ones:

  • class — gives the tag a name so CSS can style it. Like a label on a box that says "style me like a card"
  • id — a unique name so JavaScript can find it. Like a serial number — only one element can have each id
<!-- class = for styling (CSS) -->
<div class="task-card">...</div>

<!-- id = for targeting (JavaScript) -->
<div id="splitterTotal">...</div>

<!-- Both together -->
<button class="demo-btn" id="saveBtn">
  Save
</button>
Your Apps
When Claude writes something like document.getElementById('toast'), that's JavaScript reaching into the HTML to grab the element with id="toast". That's the bridge between HTML and JavaScript.
What does <div class="card" id="main"> mean?
It's one box (div) with a styling label (class="card") and a unique ID (id="main") so JavaScript can find it.

Part 3: CSS — The Clothing

CSS is how you tell the browser what things should look like.

CSS stands for "Cascading Style Sheets." Every visual detail you see — colors, spacing, fonts, rounded corners, animations — is CSS. The basic pattern is dead simple: 

selector {
  property: value;
}

That's literally it. Pick what to style (the selector), then say what to change (property) and what to change it to (value). 

/* Make all buttons blue with round corners */
button {
  background: #58a6ff;
  border-radius: 8px;
  color: white;
  padding: 10px 20px;
}

/* Make anything with class "card" have a border */
.card {
  border: 1px solid #30363d;
  border-radius: 12px;
  padding: 16px;
}

Selectors: Pointing at Things

The selector is how CSS finds what to style. Three kinds cover 95% of what you'll see:

  • button — tag name (styles ALL buttons)
  • .card — dot + class name (styles everything with class="card")
  • #toast — hash + id (styles the ONE element with id="toast")

CSS Variables: Your Apps' Theme System

This is one of the coolest patterns in your apps. Instead of writing the same color everywhere, you define it once and reference it by name: 

/* Cadence defines its colors at the top */
:root {
  --accent: #6ecfcf;     /* teal */
  --orange: #e8944c;     /* orange */
  --bg: #0c1a22;         /* dark background */
}

/* Then uses them everywhere */
button {
  background: var(--accent);  /* uses the teal */
  color: white;
}
.header {
  background: var(--bg);      /* uses the dark bg */
}

This is why changing a color in one place changes it everywhere. Claude just updates the variable at the top, and every element that references it automatically updates.

Each App Has Its Own Palette
Cadence: teal (#6ecfcf) + orange (#e8944c)
Zercher: iron gray (#8090a8) + gold (#a0946a)
Bloom: rose (#e88b93) + sky (#5ba4e6)
Roman Chariots: gold (#d0a840) + sky (#52b4c8)
Sora Signal: neo-tokyo teal (#00e5c8)
All defined as CSS variables at the top of each app's file.

The Box Model: Spacing Things Out

Every element on screen is a box. CSS controls the spacing around and inside that box with three properties:

Margin = outside space, Padding = inside space, Border = the edge 

.task-card {
  margin: 12px 0;      /* 12px space above/below */
  padding: 16px;       /* 16px space inside */
  border: 1px solid #30363d;
  border-radius: 12px; /* rounded corners */
}

Part 4: JavaScript — The Brain

This is where things actually happen.

JavaScript is the only one of the three that can do things. HTML and CSS just sit there. JavaScript responds to clicks, saves data, talks to servers, and changes what's on screen.

Variables: Storing Things

A variable is a labeled box that holds a value. Like putting a name tag on a shelf: 

// Three ways to create variables
var name = 'Alex';         // old style (still works)
let score = 42;            // modern, can change later
const MAX_TASKS = 100;     // constant, never changes

// You can change let variables
score = 43;                // fine!
// But not const
MAX_TASKS = 200;           // ERROR! Can't change const
In Your Apps
Claude usually uses const for things that don't change (like settings and functions) and let for things that do (like counters and user data). var is the old way — you'll see it in older code.

Functions: Reusable Instructions

A function is a recipe you write once and use whenever you need it. Here's a real one from Cadence: 

// This shows a popup message at the bottom
function toast(msg) {
  const el = document.getElementById('toast');
  el.textContent = msg;
  el.classList.add('visible');
  setTimeout(function() {
    el.classList.remove('visible');
  }, 3500);
}

Breaking that down:

  1. function toast(msg) — defines a function named "toast" that takes a message
  2. document.getElementById('toast') — finds the HTML element with id="toast"
  3. el.textContent = msg — puts the message text inside it
  4. el.classList.add('visible') — adds a CSS class to make it appear
  5. setTimeout(..., 3500) — waits 3.5 seconds, then hides it

Events: Responding to User Actions

JavaScript listens for things happening — clicks, typing, scrolling — and runs code in response. That's an "event": 

// When this button is clicked, run addTask
<button onclick="addTask()">Add</button>

// Or attach the listener in JavaScript
document.getElementById('saveBtn')
  .addEventListener('click', function() {
    saveData();
    toast('Saved!');
  });

The DOM: JavaScript Talking to HTML

The DOM (Document Object Model) is how JavaScript sees the HTML. Think of it as JavaScript's map of the page. The main ways JavaScript interacts with HTML: 

// FIND an element
document.getElementById('myBtn')
document.querySelector('.card')
document.querySelectorAll('.task')

// CHANGE what's inside it
element.textContent = 'New text';
element.innerHTML = '<b>Bold text</b>';

// CHANGE how it looks
element.style.color = 'red';
element.classList.add('hidden');
element.classList.remove('hidden');
element.classList.toggle('active');

// CREATE new elements
const div = document.createElement('div');
div.textContent = 'I am new!';
parent.appendChild(div);
What does document.getElementById('toast') .classList.add('visible') do?
It finds the element with id="toast" in the HTML, then adds class="visible" to it. The CSS for .visible makes it appear on screen. This is how most show/hide works in your apps.

Part 5: Reading Code Like a Map

You don't need to understand every line. Here's how to skim effectively.

When Claude writes code for you, you don't need to read every character. You need to scan for landmarks — like reading a map instead of memorizing every street name.

The Top-Down Scan

Every one of your apps follows the same structure. When you open the file, scan from top to bottom:

  1. Lines 1-20: the <head> section. Meta tags, title, font imports. Skip this — it's boilerplate.
  2. CSS section (<style> tag): all the visual styling. Look here if something looks wrong.
  3. HTML body: the actual page content — buttons, cards, inputs. Look here if something is missing.
  4. JavaScript section (<script> tag): all the logic. Look here if something doesn't work.

Reading Function Names

Claude names functions descriptively. You can usually guess what they do without reading the code inside: 

// From your real apps:
function loadData()        // loads saved data
function saveData()        // saves data
function toast(msg)        // shows a popup message
function calcPlates(w)     // calculates plates
function syncToCloud()     // syncs to the cloud
function toggleRead()      // toggles read status
function getDailyTip()     // gets the daily tip
function renderTokens()    // displays the tokens
function addTask(name)     // adds a new task
function fireGet(url)      // fires a GET request
Pro Tip
If you want to find where something happens in your code, search for the function name. If the delete button is broken, search for delete in the file. Claude can also do this for you — just say "find where the delete function is."

Comments: Claude's Notes to You

Comments are notes in the code that don't do anything — they're just explanations. They look different in each language: 

<!-- This is an HTML comment -->

/* This is a CSS comment */

// This is a JavaScript comment
/* This is also a JS comment */

Claude leaves comments to explain what sections do. They're your best friend when scanning code.

Part 6: Patterns You Already Use

These patterns appear in every one of your apps.

localStorage: Your App's Memory

localStorage is how your apps remember things between visits. It's like a tiny filing cabinet built into your browser. Every app uses it: 

// SAVE something
localStorage.setItem('tasks',
  JSON.stringify(myTasks));

// LOAD it back
const saved = localStorage.getItem('tasks');
const myTasks = JSON.parse(saved);

// DELETE it
localStorage.removeItem('tasks');

JSON.stringify converts JavaScript data into a text string (because localStorage can only store text). JSON.parse converts it back. Think of it as packing and unpacking a box.

The Big Warning
localStorage lives in your browser. If you clear Chrome's data, it's gone forever. That's why every app has cloud sync — to back up your data to Google Sheets via Apps Script.

fireGet: The Mobile Fix

This is one of the most important patterns in your apps. Normally, sending data to a server uses fetch(). But on mobile, Apps Script redirects cause fetch() to silently fail.

The fix? Create an invisible image that "loads" the URL. The browser makes the request to load the image, the server processes it, and it doesn't matter that the "image" isn't real: 

function fireGet(url) {
  const img = new Image();
  img.src = url;  // browser hits this URL
  // The server processes the request
  // We don't care about the "image" response
}

This is in Cadence, Bloom, Zercher, Crypto, SPX, Roman Chariots — every app that saves to the cloud uses this trick.

Cloud Sync: The Safety Net

Every app follows the same sync pattern:

  1. Auto-push: after every change, wait 2 seconds then save to cloud
  2. Auto-pull on load: when the app opens, check the cloud for newer data
  3. Auto-pull on resume: when you switch back to the app, check again
  4. Empty data guard: never push empty data to the cloud (prevents accidentally wiping your data)

Part 7: Your App Architecture

How all the pieces fit together.

How your apps work: GitHub Pages hosts the code, Apps Script handles data, Google Sheets stores it

Every app you have follows this exact pattern. When Claude pushes code to GitHub, GitHub Pages makes it available as a website. When you tap a button that saves data, the app calls Apps Script via fireGet(), which writes to a Google Sheet.

The Service Worker: Offline Cache

The service worker is a special file (service-worker.js) that caches your app so it works offline. The version number tells the browser when to refresh: 

// service-worker.js
const CACHE = 'app-v38';  // bump this to refresh!

This is why Claude always says "bump the service worker version" after making changes. If the version doesn't change, your phone keeps showing the old cached version — which is why you sometimes open the app and don't see the changes yet.

That Error You Saw!
Remember seeing API Error: Claude's response exceeded the 32000 output token maximum? That happens when Claude tries to write more than 32,000 tokens in one response. The fix is either writing in smaller chunks, or setting CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000 in your terminal. The default limit exists to prevent runaway responses from eating your API credits.

Part 8: The Developer Console

Every browser has a secret panel that shows you what's happening under the hood.

The console is where you can see errors, test code, and inspect your apps. On desktop Chrome: press F12 or right-click and choose "Inspect."

What the Console Shows You

  • Red errors: something broke. The message usually tells you what and which line number.
  • Yellow warnings: something might be wrong but the app still works.
  • Blue info: just informational messages.

Useful Console Commands

// See what's in localStorage
localStorage.getItem('cadence_data')

// Clear a stuck cache
localStorage.removeItem('cadence_data')

// Check if a function exists
typeof addTask  // "function" if it exists

// Run a function manually
toast('Hello from the console!')

// See all CSS variables
getComputedStyle(document.documentElement)
  .getPropertyValue('--accent')
Pro Tip
If your app is acting weird, open the console first. 90% of the time there's a red error message telling you exactly what went wrong. Tell Claude the error message and it can fix it immediately.

Part 9: Common Bugs Decoded

What error messages actually mean in plain English.

The Most Common Errors

  • Uncaught TypeError: Cannot read properties of null
    Meaning: JavaScript tried to use an element that doesn't exist. Usually means an id is misspelled or the element hasn't loaded yet.
  • Unexpected token
    Meaning: there's a typo in the code. A missing comma, bracket, or quote. The line number in the error points to where it noticed the problem.
  • CORS error
    Meaning: the browser blocked a request to another website for security. This is why your apps use fireGet() instead of regular fetch() on mobile.
  • net::ERR_FAILED
    Meaning: the network request failed. Could be no internet, wrong URL, or the server is down.
  • App shows old version after push
    Meaning: the service worker is serving cached files. The version wasn't bumped, or the cache hasn't refreshed yet. Fix: cachebust via ADB or wait.
  • Data disappeared
    Meaning: localStorage was cleared (Chrome data wipe) or cloud sync pushed empty data. That's why every app has the empty-data guard.
You push a change to Bloom but your phone still shows the old version. What's most likely the problem?
Almost always the service worker cache. The fix is bumping the version number in service-worker.js and cachebusting the phone with ADB.
